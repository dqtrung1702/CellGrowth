UAA – luồng nghiệp vụ & phân quyền

Minh họa hành trình request qua UAA: xác thực, kiểm tra quyền URL (ROLE), cấp menu trang, và áp dụng phạm vi dữ liệu (DATA).

Tóm tắt
  • User đăng nhập ➜ nhận JWT (UserId, RoleIds).
  • Middleware `enforce_url_permission` kiểm tra URL dựa trên Role ➜ Permission ➜ URLPermissions.
  • Page menu: PagePermissions quyết định hiển thị mục Home/Role/Permission/User/Datasets/Person.
  • Data scope: User có `data_permission_id` (Permission type DATA) → Set → Dataset → lọc dữ liệu.
  • Admin seed: ALL_ROLE (ROLE) + FULL_DATA (DATA) với scope `*/*/*/*`.
  • Monitor seed: chỉ `/status|/health|/ping|/static/%`, scope hẹp vào UAA.

7 bước chính

  1. 1User gửi request (mang JWT trong cookie/Authorization).
  2. 2Frontend gắn CSRF (nếu POST) và forward đến UAA.
  3. 3UAA decode JWT, lấy UserId.
  4. 4Tra UserRole ➜ RolePermission ➜ URLPermissions để quyết định 200/403.
  5. 5Tra PagePermissions ➜ trả menu khả dụng cho UI.
  6. 6Tra DataPermission (DATA) ➜ Set ➜ Dataset ➜ sinh scope.
  7. 7Scope được áp dụng vào truy vấn (WHERE / lọc client) trước khi UI nhận dữ liệu.

Cấu hình phần quyền dữ liệu (Set / Dataset)

  • Set: (Services, SetCode, SetName) – định danh phạm vi dịch vụ/miền dữ liệu.
  • Dataset: (Table, Column, Value) – điều kiện lọc chi tiết; gắn vào Set.
  • DataPermission = Permission(type=DATA) + Set → gán vào user (field data_permission_id).
  • Ý nghĩa '*':
    • Table='*' & Column='*' & Value='*' ⇒ full-access toàn bộ bảng/dịch vụ.
    • Column='*' hoặc Value='*' ⇒ wildcard cột/giá trị (ít dùng, nên hạn chế).
    • Không có scope khả dụng ⇒ từ chối (deny all).
  • Best practice: đặt Services và SetCode cụ thể; tránh dùng '*' trừ admin.

API cho service khác kiểm tra scope dữ liệu

  • POST /role_permission_api/getDataSetByUser — body { "UserId": <id> }; trả về mọi Set/Dataset DATA mà user nhận qua Role ➜ Permission(DATA).
  • POST /role_permission_api/getDatasetByPermission — body { "PermissionId": <pid> }; trả về Set/Dataset gắn với một permission DATA.
  • POST /role_permission_api/getPermissionInfo — body { "ids": [<pid>, ...] }; với permission DATA sẽ có trường DataSets.
  • POST /role_permission_api/getDatasetBySet — body { "SetId": <sid> }; trả về dataset của Set và đã enforce scope theo người gọi.
Chọn API theo mục đích
  • Audit / hiển thị phạm vi dữ liệu của user hiện tại → gọi getDataSetByUser.
  • Dựa trên mã permission (service đã biết permission id) → gọi getDatasetByPermission.
  • Batch resolve nhiều permission id một lần → gọi getPermissionInfo.
  • Đã có SetId (từ UI quản trị) cần xem chi tiết dataset → gọi getDatasetBySet.
